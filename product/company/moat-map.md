# Moat Map

## Principle

The moat cannot be only "we do simulation."

A durable Litmus moat has to compound across engine depth, workflow ownership, and customer-specific learning.

## Moat Layer 1: Verification Engine

This is the technical wedge.

Components:

- deterministic replay artifacts
- seeded fault injection
- semantic simulators
- failure-path exploration
- high-quality evidence for agent remediation

Why it matters:

- it creates memorable product wins
- it is difficult to fake with shallow wrappers
- it gives the workflow layer substance

Why it is not enough:

- engine depth alone does not own the buyer relationship
- sophisticated infra tools can still remain niche

## Moat Layer 2: Change Intelligence

This is the decision moat.

Components:

- typed diff understanding
- blast-radius inference
- risk classification
- verification recipe selection
- supported-vs-unsupported trust modeling

Why it matters:

- it makes Litmus useful on every PR, not only a handful of deep tests
- it lets the product spend verification budget intelligently
- it creates a proprietary trust model for AI-written changes

## Moat Layer 3: Organizational Learning

This is the compounding moat.

Components:

- incident-derived checks
- repeated failure pattern memory
- accepted and rejected policy outcomes
- service-specific risk models
- team-specific verification expectations

Why it matters:

- every customer gets a product that improves with their own history
- switching costs rise because the learned trust model is specific to the customer

## Moat Layer 4: Workflow Ownership

This is the distribution moat.

Components:

- local agent integration
- PR check ownership
- merge policy ownership
- deploy readiness signals
- review routing and escalation

Why it matters:

- once Litmus is in the critical path of shipping, it becomes harder to displace
- buyers pay for operational leverage, not only technical elegance

## Moat Layer 5: Ecosystem Position

This is the category moat.

Litmus should integrate with:

- coding agents
- CI systems
- GitHub and PR systems
- incident and postmortem workflows
- deployment systems

The goal is to become the trust oracle other tools call.

## What Is Not A Real Moat

- supporting every framework early
- building a large dashboard with weak underlying evidence
- generic AI review summaries
- one-off simulator cleverness without workflow adoption
- broad claims unsupported by the actual trust envelope

## Moat Summary

The durable Litmus moat is:

**verification depth + change intelligence + customer learning + workflow ownership**

That combination can become company-defining. Any one layer alone is easier to copy or out-distribute.
